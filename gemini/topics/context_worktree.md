# context: fork と Git `worktree` による高度な状態管理

## 1. 概要と概念

AIエージェントが既存の大規模コードベースに直接変更を加える際、最も恐れられるのは「動いていたメインのコードを壊してしまう」ことです。2026年には、この問題を解決し、エージェントが安全に並行して試行錯誤（トライアンドエラー）できるようにするための**状態管理技術**が大きく進化しました。

それが、**`context: fork` (エージェントの思考・メモリの分岐)** と、**Git `worktree` を用いた物理ディレクトリの分離** の組み合わせです。

### 1-1. `context: fork` とは？
会話の履歴やエージェントが「今何を考えているか（ワーキングメモリ）」そのものを Git のブランチのように分岐させる機能です。
*   「Aのライブラリを使うアプローチ」と「Bのライブラリを使うアプローチ」の両方を、同時に別の思考スレッドとして並行検証することが可能になります。
*   **A/Bテストと仮説検証の標準化**: 2026年以降は、曖昧な仕様や高リスクな変更を行う際、各分岐ごとに仮説（例：A案＝最小変更で修正、B案＝リファクタリングを含め抜本解決）と、比較基準（速度、変更量、テスト通過率など）を定めてから並行探索するのがベストプラクティスです。

### 1-2. Git `worktree` の活用
通常の `git checkout -b` では、1つのディレクトリの中身がごっそり書き換わってしまいます。これでは、複数のサブエージェントが同時に別々のコードスニペットをテストすることができません。
Gitに標準搭載されている `git worktree add` コマンドを使うと、**1つのローカルリポジトリを共有しながら、ブランチごとに完全に独立した別のフォルダを作って作業する**ことができます。これにより、AエージェントとBエージェントがファイルの競合を起こさずに同時にビルド・テストを実行可能になります。

## 3. ハンズオン：Worktreeを用いた自律的なトライアンドエラーシミュレーション

AIエージェントが背後でどのようにGitコマンドを叩いて、既存環境を汚さずに検証ループを回しているかをステップごとに体験します。ご自身のターミナル（シェル）で実行可能です。

### ステップ 1: リポジトリの準備（人間がやること）

まずはダミーのプロジェクトを作成します。

```bash
mkdir my-ai-project
cd my-ai-project
git init
echo "print('初期コード')" > main.py
git add main.py
git commit -m "Initial commit"
```

### ステップ 2: エージェントによるアプローチ分岐（Worktreeの作成）

エージェントはメインのディレクトリ (`my-ai-project`) で作業せず、使い捨ての隔離領域を作ります。

```bash
# 【エージェントのアクション】
# "feature-x" というブランチを作成し、それを親ディレクトリの "feature-x-workspace" という別フォルダに展開する
git worktree add ../feature-x-workspace -b feature-x

# これで現在のディレクトリは変わらず、1つ上の階層に安全な作業環境が生まれました
```

### ステップ 3: サブ環境でのコード変更と検証

エージェントは新しく作られた `feature-x-workspace` ディレクトリに移動し、そこでコードを変更して動作確認を行います。

```bash
# 【エージェントのアクション】
cd ../feature-x-workspace

# コードの変更（LLMが生成）
echo "print('新機能のコード！')" >> main.py

# テストの実行（検証）
python main.py
```

### ステップ 4: 比較検証と意思決定の記録

テスト結果（例えば、A案はLintエラーが出たがB案はすべてグリーンだった等）に応じて、エージェントはマージするか破棄するかを決定します。このとき、**「なぜその案を採用/棄却したのか」を必ずMarkdown形式等で記録に残す**ことで、将来の設計議論でのAIのコンテキストとして再利用します。

#### パターンA：テスト成功時（マージする）

```bash
# 【エージェントのアクション: 成功パターン】
# Decisionの記録（例: ./docs/decisions/001-feature-x.md）
echo "# Decision\n- 採用: A案\n- 理由: 受入基準を満たし、変更量が最小だったため" > decision.md

git commit -am "機能Xを実装"
cd ../my-ai-project  # メインプロジェクに戻る
git merge feature-x  # 元ブランチに変更を取り込む

# 使い終わった作業フォルダとブランチのお掃除
git worktree remove ../feature-x-workspace
git branch -d feature-x
```

#### パターンB：テスト失敗時（何もせず捨てる）

もし `python main.py` でエラーが発生し、修正も不可能だと判断した場合、エージェントはメインの環境を全く汚すことなく、この失敗した環境をゴミ箱に捨てることができます。

```bash
# 【エージェントのアクション: 失敗パターン】
cd ../my-ai-project  # メインプロジェクトに戻る

# 使い終わった（失敗した）作業フォルダを強制削除
git worktree remove ../feature-x-workspace --force
git branch -D feature-x # 失敗ブランチも削除

# メインプロジェクトは1ミリも書き換わっていないため安全！
```

## 4. この技術の意義
この `context: fork` と `worktree` の統合により、AIエージェントは**「壊してもいい安全な砂場（サンドボックス）」をミリ秒単位で無限に生成できる**ようになりました。これにより、人間のエンジニアが心理的安全性のために行っていた「とりあえず別ブランチを切って別フォルダで試す」という作業を、AIが毎秒数百回規模の高速並列処理で実行できるのが2026年以降のエージェントアーキテクチャの真髄です。

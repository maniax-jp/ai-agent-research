# 06. context forkによる並列探索

## 技術動向
`context fork` は、1つの会話文脈を分岐させ、複数案を平行評価するための設計です。2026年以降は、曖昧仕様や高リスク変更での標準運用になります。

## 設計ポイント
- 分岐ごとに仮説を明確化する（A案/B案）
- 比較基準を先に固定する（速度、安全性、変更量）
- 採用/棄却の理由を必ず記録する
- 分岐ごとに `worktree` を割り当てて物理的に分離する

## ハンズオン（2分岐比較）
1. 分岐記録ディレクトリを作成
```powershell
New-Item -ItemType Directory -Force forks/A,forks/B | Out-Null
```

2. 仮説ファイル作成
```markdown
# A案
- 最小変更で不具合修正
- 既存設計を維持
```

```markdown
# B案
- 設計を整理して再発防止
- 変更量は増える
```

3. 比較基準を定義
```markdown
- テスト通過率
- 変更ファイル数
- 性能劣化有無
- レビュー指摘数
```

4. 判定マトリクスを作成
```markdown
| Fork | Tests | Files Changed | Review Findings | Latency(ms) | Result |
|------|-------|---------------|-----------------|-------------|--------|
| A    |       |               |                 |             |        |
| B    |       |               |                 |             |        |
```

5. A/Bを別々に実装・検証（同時または順次）

6. 採用判定を記録
```markdown
# Decision
- 採用: A
- 理由: 受入基準を満たし、変更量が最小
- 棄却: B（改善余地はあるが今回スコープ外）
```

## 期待結果
- 主観ではなく比較可能な根拠で方針決定できる
- 将来の設計議論で意思決定ログを再利用できる

## 失敗しやすい点
- 比較基準を後付けにして、都合の良い案を採用してしまう
- Forkを増やしすぎて評価コストが逆転する
- 採用理由を残さず、同じ議論を繰り返す

## 次の強化
- forkごとに自動ベンチマークを実行
- 採用率の高い分岐パターンをテンプレート化
